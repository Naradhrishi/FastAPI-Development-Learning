### Piece 4: The main magic - RateLimiter bouncer at gate

```python
@app.patch("/api/v3/users/{user_id}", dependencies = [Depends(RateLimiter(times = 2, seconds = 60))],  response_model = User)
```

Break this long line:

`@app.patch("/api/v3/users/{user_id}" ...)` = This API updates a user.

`dependencies = [...]` = Before entering this classroom, you must pass these checks. It's a list, you can put many bouncers.

`Depends(RateLimiter(times = 2, seconds = 60))` = One bouncer.

Inside `RateLimiter`:

- `times = 2` = How many times allowed? 2 times.
- `seconds = 60` = In how many seconds? In 60 seconds.

So meaning: Allow only 2 times per 60 seconds.

`Depends(...)` = FastAPI, please call this bouncer automatically before my function. I don't want to call him manually.

`response_model = User` = After updating, return User.

**What bouncer does internally (you don't write this, but this is what happens):**

1. User 1 with IP `127.0.0.1` comes to PATCH `/users/3`
2. Bouncer goes to Memurai notebook, searches key: `fastapi-limiter:127.0.0.1:/api/v3/users/3:2:60`
3. Not found? He writes `1` and sets timer `expire 60 sec` and says OK go in.
4. Same IP comes again within 60 sec? He reads `1`, makes `2`, says OK.
5. Same IP comes 3rd time within 60 sec? He reads `2`, 2+1=3 which is >2, so he says STOP! Returns 429.
6. After 60 sec, Memurai auto-erases the key. So next time count starts from 1 again.

That's why you saw `retry-after: 24` - meaning 24 seconds left to erase.
